# Requirements Document

## Introduction

Migration of the GAMEC website (igamec.org) from a static HTML5 site with jQuery-based dynamic component injection to an Astro-based static site generator. The migration preserves all existing functionality, design, accessibility patterns, and URL structure while gaining the benefits of component-based architecture, build-time optimization, and modern developer experience. The site continues to deploy as static files with no server-side runtime.

## Glossary

- **Astro**: A static site generator that outputs zero-JavaScript HTML by default, supports component-based architecture, and allows opt-in client-side interactivity via "islands"
- **Astro_Layout**: A reusable `.astro` template file that wraps page content with shared HTML structure (head, header, footer, scripts)
- **Astro_Component**: A reusable `.astro` file representing a UI fragment (header, footer, navigation, card, etc.) that renders at build time
- **Astro_Page**: A `.astro` file in the `src/pages/` directory that maps to a URL route in the built site
- **Client_Island**: An interactive component that hydrates on the client using Astro's `client:*` directives, enabling JavaScript execution in the browser
- **Build_Output**: The static HTML, CSS, and asset files generated by `astro build` into the `dist/` directory
- **CSS_Module_System**: The existing modular CSS architecture using `@import` statements in `main.css` to load 23 separate concern-based CSS files
- **WAI_ARIA_Navigation**: The existing accessible navigation pattern using `aria-expanded`, keyboard handlers, and disclosure pattern for dropdown menus
- **PDF_Viewer**: The Quran PDF viewer component using PDF.js loaded via CDN with custom navigation controls
- **Receipt_Generator**: The client-side donation receipt tool that parses CSV files and generates printable receipts using localStorage for sequencing
- **Scroll_Reveal**: The IntersectionObserver-based animation system that adds `.is-visible` to elements with `.reveal` or `.reveal-stagger` classes
- **React_Island**: A React component rendered within an Astro page using Astro's `@astrojs/react` integration and hydrated on the client via `client:*` directives, enabling React's state management and lifecycle within an otherwise static page
- **Astro_React_Integration**: The official `@astrojs/react` package that enables `.jsx` and `.tsx` React components to be used inside Astro pages and components with selective client-side hydration
- **Clerk**: A third-party authentication and user management platform that provides pre-built UI components, session management, and user APIs, planned for future integration into the GAMEC site
- **Astro_Sitemap_Integration**: The official `@astrojs/sitemap` package that automatically generates an XML sitemap at build time based on the pages in `src/pages/`
- **Core_Web_Vitals**: Google's set of user-centric performance metrics — Largest Contentful Paint (LCP), Cumulative Layout Shift (CLS), and Interaction to Next Paint (INP) — used as ranking signals in search results
- **Structured_Data**: JSON-LD markup embedded in HTML pages that provides machine-readable information to search engines about the page content, organization, and relationships
- **Heading_Hierarchy**: The sequential ordering of HTML heading elements (h1 through h6) where each page has exactly one h1 and subsequent headings do not skip levels (e.g., h2 is not followed directly by h4)

## Requirements

### Requirement 1: Astro Project Initialization

**User Story:** As a developer, I want the project converted to an Astro project structure, so that I can use component-based architecture and build-time optimization.

#### Acceptance Criteria

1. THE Astro_Project SHALL include `astro` as a dependency in `package.json` with a version constraint of `>=5.0.0 <6.0.0`, and the `astro build` command SHALL complete without errors producing output in the `dist/` directory
2. THE Astro_Project SHALL include a `package.json` with scripts `"dev": "astro dev"`, `"build": "astro build"`, and `"preview": "astro preview"`, and SHALL declare a minimum Node.js engine of `>=18.17.1`
3. THE Astro_Project SHALL include a `tsconfig.json` that extends `astro/tsconfigs/strict` to enable strict TypeScript checking compatible with Astro's type system
4. THE Astro_Project SHALL include an `astro.config.mjs` file with `output: 'static'` and `site: 'https://igamec.org'` configured
5. THE Astro_Project SHALL place source files in a `src/` directory with subdirectories `pages/` for route-generating page components, `layouts/` for page layout templates, `components/` for reusable UI components, and `styles/` for global CSS files
6. THE Astro_Project SHALL place static assets (images, webfonts, manifest.json, favicon files) in a `public/` directory so they are copied to the build output without processing

### Requirement 2: Layout and Shared Component Architecture

**User Story:** As a developer, I want the shared header, footer, and page shell converted to Astro layouts and components, so that I no longer need client-side JavaScript to inject shared HTML.

#### Acceptance Criteria

1. THE Astro_Layout SHALL render the complete HTML document shell including `<!DOCTYPE html>`, `<html lang="en">`, `<head>`, and `<body class="is-preload">` tags at build time
2. THE Astro_Layout SHALL accept props for page-specific metadata (title, description, canonical URL, OG tags, Twitter Card tags, robots directive, body class, structured data) and SHALL apply default values when optional props are omitted: title defaults to "GAMEC", description defaults to the organization tagline, robots defaults to "index, follow", and OG image defaults to the site logo URL
3. THE Astro_Component for the header SHALL render the full navigation markup including dropdown menus, skip link, logo, `aria-haspopup`, and `aria-expanded` attributes at build time, and SHALL mark the current page's nav item with the `.current` class based on the page's file path
4. THE Astro_Component for the footer SHALL render the four-column footer layout (sitemap, resources, quick links, contact), social links with `target="_blank"` and `rel="noopener noreferrer"`, and copyright notice at build time
5. WHEN the site is built, THE Astro_Layout SHALL produce static HTML output that contains no `<script>` tag referencing `includes.js` and no runtime `fetch()` calls for `header.html` or `footer.html`
6. THE Astro_Layout SHALL include the noscript fallback message at the top of `<body>` using the existing `.noscript-warning` class
7. WHEN the site is built, THE Astro_Layout SHALL preserve `<script>` tags for interactive navigation behavior (desktop dropdown keyboard/hover handlers and mobile panel toggle from `main.js`) in the built output

### Requirement 3: Page Migration

**User Story:** As a developer, I want all 18 HTML pages converted to Astro pages, so that they use the shared layout and component system while preserving their content and structure.

#### Acceptance Criteria

1. THE Astro_Page files SHALL produce DOM-equivalent HTML output to the original pages for all 18 pages (index, vision, history, leadership, contact, programs, relief, sisters, youth, seniors, professionals, health, membership, donate, media, resources, matrimonial, donation-receipts), where DOM-equivalent means the same element hierarchy, tag names, text content, attributes, and class names within the `<body>` content area
2. THE Astro_Page for the homepage SHALL include all 10 sections identified by their `data-section` attribute values: `banner`, `features`, `history`, `impact`, `donate-cta`, `livestream`, `matrimonial-callout`, `resources`, `community-groups`, and `stay-connected`
3. WHEN the site is built, THE Build_Output SHALL produce one HTML file per original page at URL paths matching the original filenames (e.g., `/index.html`, `/donate.html`, `/donation-receipts.html`) so that all 18 existing URLs continue to resolve
4. THE Astro_Page for `donation-receipts` SHALL include the `<meta name="robots" content="noindex, nofollow">` directive in the rendered `<head>` via layout props
5. THE Astro_Page files SHALL preserve all `data-section` attributes, semantic HTML elements (`<main>`, `<section>`, `<nav>`, `<article>`), ARIA attributes (`aria-expanded`, `aria-selected`, `aria-controls`, `aria-labelledby`, `role`), and `visually-hidden` screen-reader text spans from the original pages
6. THE Astro_Page files SHALL load page-specific scripts only on their corresponding pages: `donation-receipts.js` on the donation-receipts page and `quran-viewer.js` on the resources page, in addition to the shared script bundle
7. THE Astro_Page files SHALL preserve the `data-page` attribute on the `<body>` element and any page-specific body classes (e.g., `homepage`, `no-sidebar`) from the original HTML pages so that CSS and JS selectors continue to function

### Requirement 4: CSS Migration

**User Story:** As a developer, I want the existing CSS module system preserved and integrated into Astro's build pipeline, so that all styles continue to work without visual regression.

#### Acceptance Criteria

1. THE Astro_Project SHALL import the existing `main.css` entry point (which `@import`s all 23 CSS modules) and `media.css` as global stylesheets, preserving the cascade order defined by the `@import` sequence in `main.css`
2. THE CSS_Module_System SHALL retain all CSS source files unmodified, preserving CSS custom properties, BEM naming, and the cascade order so that the build output maintains the same selector specificity and declaration order as the source
3. THE Astro_Project SHALL serve Font Awesome from local webfont files in `public/assets/webfonts/` with the existing `fontawesome-all.min.css` stylesheet, and font file paths referenced in the stylesheet SHALL resolve correctly to the served webfont files (fa-brands-400, fa-regular-400, fa-solid-900, fa-v4compatibility in woff2 and ttf formats)
4. WHEN the site is built, THE Build_Output SHALL produce CSS that renders identically to the current site across all 5 responsive breakpoints (default >1680px, xlarge 1281–1680px, large 981–1280px, medium 737–980px, small ≤736px), verified by confirming no differences in computed layout, colors, typography, or spacing on each page
5. THE Astro_Project SHALL preserve the Google Fonts imports for Roboto (weights 400, 500, 700) and Playfair Display (weights 400, 700) with `display=swap` as defined in `_imports-fonts.css`
6. IF the Astro build fails to resolve any of the 23 CSS module `@import` paths or the Font Awesome stylesheet path, THEN THE Build_Process SHALL produce a build error indicating which file path failed to resolve

### Requirement 5: JavaScript Migration

**User Story:** As a developer, I want the client-side JavaScript migrated to work within Astro's architecture, so that interactive features continue to function while eliminating unnecessary scripts.

#### Acceptance Criteria

1. WHEN the site is built, THE Build_Output SHALL NOT include `includes.js` since header/footer injection is handled at build time by Astro components
2. THE Astro_Project SHALL preserve the WAI_ARIA_Navigation initialization logic as a client-side script that executes after the DOM is ready, including: desktop dropdowns that open on mouseenter and close on mouseleave with a 200ms delay, keyboard navigation (Enter/Space/ArrowDown to open, ArrowUp/ArrowDown to move between items, Escape to close), `aria-expanded` toggling on parent links, and a mobile slide-in panel toggled by a button with `aria-expanded` and `aria-label` state management
3. THE Astro_Project SHALL preserve the Scroll_Reveal IntersectionObserver logic as a client-side script that observes elements with `.reveal` or `.reveal-stagger` classes, adds `.is-visible` when 15% of the element enters the viewport (threshold 0.15), unobserves each element after first intersection, and falls back to adding `.is-visible` immediately if IntersectionObserver is not supported
4. THE Astro_Project SHALL preserve the preload state pattern where the `<body>` element has the `.is-preload` class removed 100ms after the window load event fires
5. THE Astro_Page for resources SHALL load PDF.js via CDN and initialize the PDF_Viewer preserving all 14 existing functions: page navigation (goToPage, clampPage), zoom controls (doZoom, resetZoom, clampZoom, adjustZoom), surah table of contents (buildToc, toggleToc, getCurrentSurah), localStorage state persistence (saveState, loadState), page rendering (renderPage), and error display (showError)
6. THE Astro_Page for donation-receipts SHALL load the Receipt_Generator script preserving all existing functions: CSV parsing (parseCSVLine, parseCSV), receipt generation (generateReceipt, formatReceiptNumber), form validation (validateForm, clearFormErrors, showFormErrors), localStorage sequencing (getNextSequence, setSequence, isLocalStorageAvailable, updateNextReceiptDisplay), preview rendering (renderPreviewTable, filterByYear), and utilities (formatCurrency, escapeHTML, buildMailtoLink, initReceiptGenerator)
7. THE Astro_Project SHALL eliminate the dependency on `jquery.min.js`, `browser.min.js`, `breakpoints.min.js`, and `util.js` by rewriting navigation logic in vanilla JavaScript, including replacing jQuery's `.panel()` and `.navList()` plugin calls with equivalent vanilla implementations
8. IF jQuery removal causes any of the following: dropdown menus fail to open/close, keyboard navigation stops responding, mobile panel fails to slide in/out, or ARIA attributes stop updating, THEN THE Astro_Project SHALL retain jQuery as a compatibility layer until the failing behavior is reimplemented in vanilla JavaScript
9. THE Astro_Project SHALL preserve the active navigation highlighting logic that compares the current page filename against nav link `href` values and applies a `.current` class to the matching `<li>` element, handled either at build time by Astro or as a client-side script

### Requirement 6: Active Navigation Highlighting

**User Story:** As a developer, I want the current page highlighted in the navigation, so that users can see which page they are on without client-side filename comparison logic.

#### Acceptance Criteria

1. THE Astro_Component for the header SHALL accept the current page path as a string prop and apply the `.current` class to the `<li>` whose child `<a>` href matches the current page filename (the segment after the last `/`, ignoring query strings and hashes) at build time
2. WHEN the current page path matches a navigation link nested within a dropdown submenu, THE Astro_Component SHALL apply `.current` to both the matching submenu `<li>` and its parent dropdown `<li>`
3. WHEN the current page path matches a top-level navigation link that is not inside a dropdown, THE Astro_Component SHALL apply `.current` only to that top-level `<li>`
4. IF the current page path does not match any navigation link href, THEN THE Astro_Component SHALL render the navigation with no `.current` class applied to any `<li>`

### Requirement 7: SEO and Metadata Preservation

**User Story:** As a developer, I want all SEO metadata, structured data, and social sharing tags preserved in the migrated site, so that search engine rankings and social previews are unaffected.

#### Acceptance Criteria

1. THE Astro_Layout SHALL render Open Graph tags (og:title, og:description, og:image, og:url, og:type, og:site_name, og:locale) for every page, where og:title and og:description use page-specific values passed as props, og:image uses an absolute URL, og:url matches the canonical URL, og:type is "website", og:site_name is "GAMEC", and og:locale is "en_US"
2. THE Astro_Layout SHALL render Twitter Card tags (twitter:card, twitter:title, twitter:description, twitter:image) for every page, where twitter:card is "summary", twitter:title and twitter:description match the corresponding Open Graph values, and twitter:image uses an absolute URL
3. THE Astro_Layout SHALL render a canonical URL link element for every page using the production domain (https://igamec.org) combined with the page path, matching the URL pattern used in the current static site
4. THE Astro_Page for the homepage SHALL include a JSON-LD block with @type "NonprofitOrganization" containing at minimum: name, url, logo, description, telephone, and address properties, and a second JSON-LD block with @type "WebSite" containing at minimum: name, url, and description properties
5. THE Astro_Layout SHALL render the manifest.json link, two favicon link elements (one ICO format with sizes="any", one PNG format with type="image/png" and sizes="32x32"), and one apple-touch-icon link element for every page
6. THE Astro_Layout SHALL include one dns-prefetch hint for fonts.googleapis.com, one preconnect hint for fonts.googleapis.com, and one preconnect hint with crossorigin attribute for fonts.gstatic.com
7. IF a page does not provide a title or description prop, THEN THE Astro_Layout SHALL fall back to the site-wide default values ("GAMEC" for title, and the organization-level description for description) so that no page renders empty metadata tags

### Requirement 8: Image Optimization

**User Story:** As a developer, I want images optimized during the build process, so that page load performance improves without manual image compression.

#### Acceptance Criteria

1. THE Astro_Project SHALL use Astro's built-in image optimization for content images (JPG, PNG) referenced in `<img>` tags to generate WebP format with a quality setting between 70 and 80
2. THE Astro_Project SHALL preserve `loading="lazy"` on all images that already have the attribute in the source HTML
3. THE Astro_Project SHALL preserve all existing `alt` text attributes on images without modification
4. THE Astro_Project SHALL keep hero banner images as CSS backgrounds where they are currently implemented that way (homepage `#banner-wrapper`) and serve those images from the `public/` directory without build-time format conversion
5. IF an image is referenced in CSS (background-image), THEN THE Astro_Project SHALL serve it from the `public/` directory without build-time processing or format conversion
6. THE Astro_Project SHALL pass through SVG files without raster conversion or quality reduction
7. IF a content image referenced in an `<img>` tag cannot be resolved at build time, THEN THE Astro_Project build SHALL fail with an error message indicating the missing file path

### Requirement 9: Build Output and Deployment Compatibility

**User Story:** As a developer, I want the build output to be deployable as static files to the same hosting environment, so that the deployment process remains unchanged.

#### Acceptance Criteria

1. WHEN `astro build` is run, THE Build_Output SHALL produce a `dist/` directory containing only static HTML, CSS, JavaScript, and asset files with no server-side runtime requirements (no Node.js server, no serverless functions, no package.json dependencies needed at deploy time)
2. THE Build_Output SHALL produce `.html` file extensions matching the current URL structure (e.g., `dist/donate.html`, `dist/contact.html`, `dist/index.html`) so that all 18 existing page URLs continue to resolve without server-side rewrites or redirects
3. THE Build_Output SHALL include all files from `public/` with their directory structure preserved at the root of `dist/` (e.g., `public/images/cover.png` → `dist/images/cover.png`, `public/assets/webfonts/` → `dist/assets/webfonts/`, `public/manifest.json` → `dist/manifest.json`)
4. THE Astro_Project SHALL include a `.gitignore` entry for the `dist/` directory and `node_modules/`
5. IF the URL structure changes from `.html` extensions to clean URLs in a future update, THEN THE Astro_Project SHALL document the redirect rules needed for the hosting provider in a `DEPLOYMENT.md` file at the project root
6. WHEN `astro build` is run, THE Build_Output SHALL produce exactly 18 HTML files corresponding to the current page inventory (index, vision, history, leadership, contact, programs, relief, sisters, youth, seniors, professionals, health, membership, donate, media, resources, matrimonial, donation-receipts)

### Requirement 10: Development Experience

**User Story:** As a developer, I want a modern development workflow with hot reload and clear project organization, so that future changes are faster and less error-prone.

#### Acceptance Criteria

1. WHEN a developer runs `astro dev`, THE Astro_Project SHALL start a local development server with hot module replacement that reflects saved changes to `.astro`, `.css`, and `.js` files in the browser without a full page reload
2. THE Astro_Project SHALL extract repeated or page-specific UI fragments into reusable Astro components in `src/components/`, including at minimum: Header, Footer, LivestreamCard, ProgramCard, ImpactStats, DonationMethods, and CommunityGroupCard
3. THE Astro_Project SHALL include a README.md containing at minimum: a project overview, the `src/` directory structure with descriptions of `pages/`, `layouts/`, `components/`, and `styles/` directories, a table of available npm scripts (`dev`, `build`, `preview`), and step-by-step instructions for adding a new page using the shared layout
4. THE Astro_Project SHALL configure path aliases in `tsconfig.json` mapping at minimum `@components/*` to `src/components/*`, `@layouts/*` to `src/layouts/*`, and `@styles/*` to `src/styles/*`
5. THE Astro_Project SHALL use PascalCase filenames for component files in `src/components/` (e.g., `LivestreamCard.astro`, `ProgramCard.astro`) to distinguish them from page files which use lowercase

### Requirement 11: Accessibility Preservation

**User Story:** As a developer, I want all accessibility features preserved in the migrated site, so that screen reader and keyboard-only users have the same experience.

#### Acceptance Criteria

1. THE Astro_Component for the header SHALL preserve the skip-to-content link (`<a href="#main-wrapper" class="skip-link visually-hidden">`) and every page layout SHALL contain a corresponding target element with `id="main-wrapper"` so the skip link resolves to a valid destination
2. THE Astro_Component for the header SHALL preserve all `role`, `aria-label`, `aria-haspopup`, and `aria-expanded` attributes on navigation elements, including `aria-label="Main navigation"` on the `<nav>` element and `aria-haspopup="true"` and `aria-expanded="false"` on each parent dropdown link
3. THE Astro_Project SHALL preserve the `.visually-hidden` utility class (with the clip-based hiding technique) and its usage for screen reader announcements on icon-only buttons (social media links in footer) and external links (new-tab indicator text)
4. THE Astro_Project SHALL preserve the WAI_ARIA_Navigation keyboard interaction pattern: Enter, Space, or ArrowDown on a parent link opens the dropdown and moves focus to the first dropdown item; ArrowDown/ArrowUp navigates between dropdown items; Escape closes the dropdown and returns focus to the parent link; Tab closes the dropdown and moves focus to the next focusable element in document order
5. THE Astro_Project SHALL preserve `role="banner"` on the header element, `role="navigation"` on the nav element, and `role="contentinfo"` on the footer element
6. THE Astro_Project SHALL preserve the mobile navigation toggle button with `aria-label` toggling between "Open menu" and "Close menu" and `aria-expanded` toggling between "false" and "true" to reflect the panel's open/closed state
7. WHEN the skip-to-content link receives focus, THE Astro_Component for the header SHALL display the link visibly (using the `.skip-link:focus` styles) so keyboard users can see and activate it

### Requirement 12: Matrimonial Page Interactive Features

**User Story:** As a developer, I want the matrimonial page accordion preserved as a client-side interactive component, so that the FAQ section remains collapsible and accessible.

#### Acceptance Criteria

1. THE Astro_Page for matrimonial SHALL render all accordion sections as native `<details>` elements with `<summary>` children at build time, with all panels defaulting to the collapsed state (no `open` attribute)
2. THE Astro_Page for matrimonial SHALL use native `<details>/<summary>` elements for toggle behavior, requiring no custom JavaScript for expanding and collapsing panels
3. WHEN a user presses Enter or Space while a `<summary>` element has focus, THE accordion component SHALL toggle the corresponding panel between expanded and collapsed states, and focus SHALL remain on the activated `<summary>` element
4. THE accordion component SHALL rely on the native `<details>` element semantics to communicate expanded and collapsed state to assistive technologies without requiring explicit `aria-expanded` or `aria-controls` attributes


### Requirement 13: React Islands for Interactive Components

**User Story:** As a developer, I want to use React islands for complex interactive components, so that I can leverage React's state management and component model where client-side interactivity demands it while keeping the rest of the site as zero-JS static HTML.

#### Acceptance Criteria

1. THE Astro_Project SHALL include `@astrojs/react`, `react`, and `react-dom` as dependencies in `package.json` and register the React integration in `astro.config.mjs`
2. THE Astro_Project SHALL place React island components in a `src/components/react/` subdirectory using `.tsx` file extensions with PascalCase filenames (e.g., `QuranViewer.tsx`)
3. THE Astro_Page for resources SHALL render the PDF_Viewer as a React_Island using `client:visible` hydration so that the component loads React only when the viewer scrolls into the viewport, preserving all 14 existing viewer functions (page navigation, zoom controls, surah table of contents, localStorage state persistence, page rendering, and error display)
4. WHEN a component requires complex client-side state management (multi-step forms, real-time filtering, or stateful UI with multiple interdependent controls), THE Astro_Project SHALL implement the component as a React_Island rather than vanilla JavaScript
5. THE React_Island components SHALL use `client:visible` as the default hydration directive unless immediate interactivity on page load is required, in which case `client:load` SHALL be used
6. THE React_Island components SHALL not increase the initial page load bundle for pages that do not use React islands — Astro's per-page bundling SHALL ensure React is only loaded on pages containing React_Island components
7. IF a React_Island fails to hydrate due to a JavaScript error, THEN THE React_Island SHALL render its server-side fallback content (the static HTML produced at build time) so that users see meaningful content rather than a blank area

### Requirement 14: Future Clerk Authentication Compatibility

**User Story:** As a developer, I want the Astro project structured for future Clerk authentication integration, so that adding user login, protected pages, and member-only content requires minimal architectural changes later.

#### Acceptance Criteria

1. THE Astro_Project SHALL include the Astro_React_Integration as a prerequisite for Clerk's React-based UI components (sign-in, sign-up, user profile), satisfying Clerk's dependency on React at the framework level
2. THE Astro_Project SHALL reserve the URL paths `/sign-in` and `/sign-up` by placing placeholder Astro_Page files at `src/pages/sign-in.astro` and `src/pages/sign-up.astro` that render a "Coming Soon" message, ensuring these routes are available for Clerk's authentication pages without future URL conflicts
3. THE Astro_Project SHALL organize components in a structure that accommodates auth-aware UI by reserving a `src/components/auth/` directory for future authentication components (sign-in button, user menu, protected route wrapper)
4. THE Astro_Project SHALL document in README.md under a "Future: Authentication" section that Clerk is the planned authentication provider, listing the prerequisites already in place (React integration, reserved routes, auth component directory) and the remaining steps to complete integration (install `@clerk/astro`, add environment variables, wrap layouts with ClerkProvider)
5. WHEN Clerk integration is implemented in the future, THE Astro_Project structure SHALL allow adding authentication without modifying the existing Astro_Layout, page routing, or React_Island architecture — Clerk components SHALL integrate as additional React islands within the existing component system
6. THE Astro_Project SHALL NOT install Clerk packages (`@clerk/astro`, `@clerk/react`) or configure Clerk environment variables until authentication is actively being implemented, avoiding unused dependencies and configuration drift

### Requirement 15: SEO Optimization and Enhancement

**User Story:** As a developer, I want the Astro migration to introduce SEO improvements beyond preserving existing metadata, so that the site gains better search engine visibility, faster page speeds, and richer search result appearances enabled by Astro's static-first architecture.

#### Acceptance Criteria

1. THE Astro_Project SHALL include `@astrojs/sitemap` as a dependency registered in `astro.config.mjs`, and WHEN `astro build` is run, THE Astro_Sitemap_Integration SHALL generate a valid XML sitemap at `dist/sitemap-index.xml` containing URLs for all public pages (excluding pages with `noindex` robots directives such as donation-receipts)
2. THE Astro_Project SHALL generate a `robots.txt` file in the Build_Output that references the sitemap URL (`Sitemap: https://igamec.org/sitemap-index.xml`), allows all user agents (`User-agent: *`), and disallows no paths
3. WHEN the site is built, THE Build_Output SHALL achieve a Largest Contentful Paint (LCP) of 2.5 seconds or less, a Cumulative Layout Shift (CLS) of 0.1 or less, and an Interaction to Next Paint (INP) of 200 milliseconds or less on all pages when measured on a simulated mobile connection (4G throttling) using Lighthouse or equivalent tooling
4. THE Astro_Project SHALL produce pages with zero render-blocking JavaScript by default (Astro's zero-JS output), loading client-side scripts only on pages that require interactivity (resources, donation-receipts) via deferred or async loading strategies
5. THE Astro_Project SHALL enforce a valid Heading_Hierarchy on every page: exactly one `<h1>` element per page, no skipped heading levels within the page content, and heading levels that descend sequentially (h1 → h2 → h3) within each content section
6. THE Astro_Layout SHALL use semantic HTML elements for page structure on every page: `<header>` for the site header, `<nav>` for navigation, `<main>` for primary content, `<section>` with descriptive `aria-label` or `aria-labelledby` for content sections, `<article>` for self-contained content blocks, and `<footer>` for the site footer
7. THE Astro_Project SHALL include Structured_Data (JSON-LD) on all program pages (programs, relief, sisters, youth, seniors, professionals, health) with @type "NonprofitOrganization" and a "department" or "subOrganization" property describing the specific program, and on the contact page with @type "NonprofitOrganization" including full ContactPoint data (telephone, email, contactType)
8. THE Astro_Project SHALL generate optimized image assets at build time with appropriate `width` and `height` attributes on all `<img>` elements to prevent layout shift, and SHALL include `fetchpriority="high"` on above-the-fold hero images to improve LCP
9. THE Astro_Project SHALL inline critical CSS required for above-the-fold content rendering and defer non-critical stylesheets so that the First Contentful Paint is not blocked by full stylesheet loading
10. THE Astro_Project SHALL produce Build_Output with minified HTML, CSS, and JavaScript files, and SHALL apply content-hash-based filenames to processed CSS and JS assets to enable long-term browser caching with cache-busting on content changes
11. IF a page contains a `<meta name="robots" content="noindex, nofollow">` directive, THEN THE Astro_Sitemap_Integration SHALL exclude that page from the generated sitemap
